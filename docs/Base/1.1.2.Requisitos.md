# Requisitos de Software

## Introdução

Este documento especifica os requisitos de software do sistema G7_MonitoreSeuTreino, derivados a partir dos seguintes artefatos do projeto: documento [5W2H](Base/1.2.ArtefatoGeneralista.md), [Visão do Produto](Base/1.1.1.VisaoProduto.md), [Léxico Ampliado da Linguagem](Base/1.2.ArtefatoGeneralista.md), Mapa Mental, Benchmark de Concorrentes, Diagrama de Ishikawa, Perfil de Cliente Ideal (ICP) e Rich Pictures elaborados por cada membro da equipe. Está organizado em glossário de termos, referência aos objetivos específicos, regras de negócio, requisitos funcionais organizados por módulo com rastreabilidade e requisitos não funcionais mensuráveis com critérios de teste explícitos.

## Glossário de Termos

Os termos a seguir são utilizados ao longo de todo o documento e seguem as definições do [Léxico Ampliado da Linguagem](Base/1.2.ArtefatoGeneralista.md) do projeto:

| Termo | Definição |
| ----- | --------- |
| Usuário | Pessoa que utiliza o sistema para planejar e registrar sua rotina de treino semanal. |
| Rotina | Ficha de treino semanal que organiza exercícios e parâmetros planejados por dia ou divisão. |
| Sessão | Registro da execução real de um treino em uma data, contendo os valores efetivamente realizados. |
| Exercício | Movimento físico cadastrado no sistema, identificado por nome e grupo muscular. |
| Grupo muscular | Classificação anatômica do exercício. |
| Parâmetros planejados | Conjunto de valores definidos na rotina para orientar a execução: séries alvo, repetições alvo e observações. |
| Série | Conjunto de repetições de um exercício executadas continuamente antes do descanso. |
| Repetições | Quantidade de execuções completas de um exercício dentro de uma série. |
| Carga | Peso ou resistência utilizada em um exercício durante uma série. |
| Observação | Texto livre associado a uma sessão ou exercício para registrar detalhes adicionais. |
| Perfil | Combinação de gênero e nível de experiência do usuário, resultante do onboarding. |
| Inativar | Tornar exercícios ou rotinas indisponíveis para novos usos, preservando o histórico de sessões já registradas. |
| Excluir | Remoção definitiva de sessões ou contas de usuário, incluindo todos os dados associados. |

## Referência aos Objetivos Específicos

Os requisitos funcionais são rastreados aos objetivos específicos da [Visão do Produto](Base/1.1.1.VisaoProduto.md) e às características da solução correspondentes:

| Código | Objetivo Específico |
| ------ | ------------------- |
| OE1 | Introduzir organização no planejamento do treino semanal para usuários que ainda não possuem estrutura definida, oferecendo uma forma simples e padronizada de montar e visualizar sua rotina de exercícios. |
| OE2 | Eliminar a perda de informações sobre os treinos realizados, garantindo que o usuário tenha um registro confiável e simplificado de cada sessão executada. |
| OE3 | Possibilitar o acompanhamento da evolução do usuário ao longo do tempo, permitindo a recuperação e comparação de dados de treinos anteriores. |
| OE4 | Aumentar a visibilidade da constância semanal do usuário, tornando clara a regularidade dos treinos e incentivando a manutenção da disciplina. |
| OE5 | Reduzir a barreira de entrada para usuários que não sabem por onde começar, assegurando que cada pessoa inicie com um treino compatível com seu gênero e nível de experiência. |

## Regras de Negócio

As regras de negócio definem políticas, restrições e definições globais do sistema. São referenciadas nos requisitos funcionais quando aplicáveis. Não descrevem funcionalidades nem comportamentos de interface.

| Código | Regra | Descrição |
| ------ | ----- | --------- |
| RN01 | Definição de semana | A semana para fins de resumo semanal é considerada de segunda-feira 00:00 até domingo 23:59, no fuso horário configurado pelo sistema. |
| RN02 | Contagem de sessões | Cada sessão concluída conta como uma unidade no resumo semanal, independentemente da sua duração ou quantidade de exercícios. Múltiplas sessões no mesmo dia são contabilizadas individualmente. |
| RN03 | Definição de constância semanal | A constância semanal é composta por duas métricas: a quantidade total de sessões concluídas na semana e a quantidade de dias distintos com pelo menos uma sessão. |
| RN04 | Associação sessão-rotina | A associação entre uma sessão e uma rotina é opcional. O usuário pode registrar sessões avulsas sem vínculo a nenhuma rotina. |
| RN05 | Preservação do histórico | Exercícios e rotinas inativados pelo usuário permanecem visíveis nas sessões já registradas. O histórico não é afetado por inativações. |
| RN06 | Independência da sessão registrada | Quando uma sessão é criada a partir de uma rotina com dados pré-preenchidos, alterações feitas pelo usuário na sessão não afetam a rotina original. A sessão mantém apenas a referência de origem para rastreabilidade. |
| RN07 | Unicidade de rotina ativa | Apenas uma rotina pode estar marcada como ativa por vez. A rotina ativa é sugerida como padrão ao iniciar o registro de uma nova sessão. |
| RN08 | Níveis de experiência | O sistema reconhece três níveis de experiência: iniciante, intermediário e avançado. O nível é atribuído ao usuário com base nas respostas do questionário de onboarding. |
| RN09 | Classificação por gênero | O sistema classifica o usuário como masculino ou feminino para fins de adequação da montagem de treino. Essa informação é coletada durante o onboarding. |
| RN10 | Montagem de treino por perfil | A combinação de gênero e nível de experiência determina qual modelo de treino pré-definido será sugerido ao usuário após a conclusão do onboarding. O usuário pode adotar, ajustar, substituir ou recusar livremente o treino sugerido. |
| RN11 | Estados da sessão | Uma sessão possui o estado de concluída quando foi salva com sucesso pelo usuário. Apenas sessões concluídas são contabilizadas no histórico e no resumo semanal. O conceito de sessão em rascunho está fora do escopo do MVP. |
| RN12 | Dados pessoais | O sistema coleta e armazena dados pessoais do usuário estritamente necessários para o funcionamento das funcionalidades: nome, e-mail, senha, gênero e nível de experiência. A exclusão da conta deve remover todos os dados pessoais associados ao usuário. |

## Requisitos Funcionais

Os requisitos funcionais descrevem os comportamentos específicos que o sistema deverá implementar. Estão organizados por módulo, orientados a comportamento, com referências explícitas às regras de negócio aplicáveis e rastreabilidade ao objetivo específico e à característica da solução correspondente. Requisitos transversais ao sistema, como autenticação e gestão de conta, são marcados como "Transversal" na coluna de rastreabilidade.

## Requisitos Não Funcionais

Os requisitos não funcionais descrevem as qualidades e restrições que o sistema deve atender. Todos são mensuráveis e incluem critério de teste explícito. Estão organizados por categoria de qualidade de software.

<details open>
<summary><strong>Usabilidade</strong></summary>

| Código | Nome | Descrição e Critério de Teste |
| ------ | ---- | ----------------------------- |
| RNF-U01 | Navegação intuitiva | O fluxo principal — criar rotina, registrar sessão, consultar resumo — deve ser acessível em no máximo 3 níveis de navegação a partir da tela inicial. Critério de teste: validado por teste de usabilidade com no mínimo 5 usuários, onde 80% devem completar o fluxo sem assistência em até 5 minutos. |
| RNF-U02 | Feedback visual | O sistema deve fornecer feedbacks visuais claros e imediatos para todas as ações do usuário, incluindo confirmações, erros e indicadores de carregamento. Critério de teste: 100% das ações de criação, edição, inativação e exclusão devem exibir mensagem de confirmação ou erro, verificado por checklist funcional em cada release. |
| RNF-U03 | Design consistente | A interface deve seguir um padrão visual único em cores, tipografia, espaçamentos e componentes. Critério de teste: auditoria visual comparando todas as telas com o guia de estilo, com zero desvios permitidos. |
| RNF-U04 | Interface responsiva | O sistema deve renderizar corretamente em viewports de 360px a 1920px, sem rolagem horizontal, com elementos interativos de no mínimo 44x44 pixels. Critério de teste: verificado em Chrome DevTools nos breakpoints 360px, 768px e 1920px para cada tela. |
| RNF-U05 | Simplicidade no registro | O registro de uma sessão com rotina vinculada não deve exigir mais de 3 interações do usuário: selecionar rotina, preencher execução e salvar. Critério de teste: contagem de cliques em teste com 5 usuários, onde 80% devem concluir em até 3 interações. |
| RNF-U06 | Onboarding conciso | O questionário de onboarding deve ser completável em no máximo 2 minutos e não deve exceder 5 perguntas. Critério de teste: cronometragem com 5 usuários em primeiro acesso, onde 100% devem concluir dentro do tempo. |

</details>

<details>
<summary><strong>Confiabilidade</strong></summary>

| Código | Nome | Descrição e Critério de Teste |

| RNF-C02 | Backup de dados | O sistema deve realizar backups automáticos diários dos dados de usuários, rotinas, sessões e histórico, com restauração completa possível em no máximo 4 horas. Critério de teste: simulação de restauração em ambiente de homologação a cada release major. |
| RNF-C03 | Consistência de dados | Edições e exclusões em sessões devem refletir no histórico e no resumo semanal em no máximo 5 segundos, sem recarregamento manual. Critério de teste: medição do tempo entre confirmação da ação e atualização visível na interface em 10 operações consecutivas. |
