# Backlog do Produto

Este documento apresenta o backlog do produto G7_MonitoreSeuTreino, estruturado hierarquicamente em Épicos, Features, User Stories, Tasks e Critérios de Aceitação. Cada User Story está vinculada ao requisito funcional de origem. Cada Task representa uma unidade de trabalho necessária para cumprir a User Story, e seus Critérios de Aceitação definem as condições objetivas para aceite da task. O backlog foi derivado do documento de requisitos de software, mantendo rastreabilidade aos objetivos específicos e às regras de negócio.

## Hierarquia do Backlog

| Nível | Código | Descrição |
| ----- | ------ | --------- |
| **Épico** | E01, E02, ... | Agrupamento de alto nível que representa uma grande capacidade ou domínio funcional do sistema. Cada épico reúne features relacionadas a um mesmo objetivo de negócio. |
| **Feature** | F01, F02, ... | Funcionalidade entregável que compõe um épico. Cada feature agrupa user stories que, juntas, entregam uma capacidade completa ao usuário. |
| **User Story** | US01, US02, ... | Necessidade do usuário descrita no formato "Como... quero... para que...". Cada user story está vinculada a um requisito funcional de origem e representa um comportamento esperado do sistema do ponto de vista do usuário. |
| **Task** | T01, T02, ... | Unidade de trabalho técnica necessária para implementar uma user story. As tasks são divididas entre backend e frontend, representando o trabalho concreto de desenvolvimento. |
| **Critério de Aceitação** | Numerados por task | Condição objetiva e verificável que define quando uma task está concluída. Os critérios referenciam regras de negócio e requisitos não funcionais aplicáveis, garantindo que cada entrega atenda tanto aos requisitos funcionais quanto aos padrões de qualidade do sistema. |
